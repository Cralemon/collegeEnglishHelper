# 用 Agent 开发一个桌面应用：思路、步骤与踩坑记录

> 这不是一个"AI 帮我写代码"的故事，而是一个"我如何指挥 AI 写代码"的记录。

---

## 背景

我要做一个大学英语翻译练习的桌面应用。技术栈：Next.js 16 + React 19 + Tauri v2。

我选择用 AI Agent（Claude Code）作为主要开发工具。这篇文章记录的是**我作为指挥者的思路**，而不是 Agent 的内部工作细节。

---

## 第一阶段：项目规划（人做）

在让 Agent 动手之前，我先自己完成了完整的规划文档（`FINAL_PLAN.md`）。这一步**必须由人来做**，原因：

1. **Agent 不了解你的需求全貌**。它可以帮你写代码，但不能替你决定"这个 App 到底要做什么"。
2. **规划文档是 Agent 的指令集**。越清晰的文档，Agent 的输出越可控。
3. **避免返工**。口头描述容易遗漏，写成文档的过程本身就是需求验证。

我的规划文档包含：
- 产品定位与核心循环
- 功能需求（按页面拆分，每项标注优先级 P0/P1/P2）
- 技术架构选型（每个选择附带理由）
- 设计规范（色彩、字体、间距、响应式断点）
- 数据模型（TypeScript 接口定义）
- 开发计划（10 个 Step，每步有明确交付物和验收标准）

**关键心得**：规划文档的质量直接决定 Agent 的产出质量。模糊的输入 = 模糊的输出。

---

## 第二阶段：环境初始化（Agent 做）

有了规划文档后，第一个 Agent 任务是项目初始化。

**我给出的指令**（大意）：
> 按照 FINAL_PLAN.md 的 Step 1，初始化 Next.js + Tailwind CSS 4 + Tauri v2 项目。

**Agent 做了什么**：
- `create-next-app` 初始化
- 配置 Tailwind CSS 4
- `tauri init` 初始化 Tauri
- 配置 `next.config.ts`（静态导出）
- 设置 `.gitignore`

**我需要检查的**：
- 版本是否符合要求（Next.js 16，不是 15）
- 配置是否正确（`output: 'export'` 对 Tauri 必需）
- `.gitignore` 是否覆盖了所有需要忽略的文件

**踩坑**：Agent 默认生成的 `next.config.ts` 没有 `output: 'export'`，这是 Tauri 静态打包必需的。如果我不检查，后续打包会失败。

## 第三阶段：用 Agent 的原则（持续）

经过几轮实践，我总结了几条核心原则：

### 原则 1：先规划，后执行

不要直接说"帮我做一个翻译练习 App"。先自己写好规划，再让 Agent 按步骤执行。

```
❌ "帮我做一个翻译练习 App"
✅ "按照 FINAL_PLAN.md 的 Step 2，实现设计系统和字体配置"
```

### 原则 2：每步验收，不要一口气跑完

10 个 Step，每完成一个就验收。验收标准在规划文档里已经写好。

为什么要验收？
- Agent 可能误解需求
- Agent 可能遗漏细节（比如忘了 `output: 'export'`）
- 早期发现问题，修复成本低

### 原则 3：让 Agent 读文档，不要假设它知道

这个项目用的是 Next.js 16，Agent 的训练数据可能还停留在 15。所以我专门写了 `AGENTS.md`，告诉 Agent：

> 在写任何代码之前，先读 `node_modules/next/dist/docs/` 里的指南。

**永远不要假设 Agent 了解你使用的具体版本的 API。**

### 原则 4：类型定义先行

在写任何业务代码之前，先把 TypeScript 类型定义好。这有两个好处：
1. Agent 写代码时有明确的类型约束，减少错误
2. 类型定义本身就是文档

### 原则 5：Git 是你的安全网

每完成一个有意义的改动就提交。如果 Agent 的改动有问题，`git revert` 比手动修复快得多。

我使用的提交规范：
```
feat(scope): 新功能
fix(scope): 修复
chore: 构建/工具变更
docs: 文档更新
```

## Agent 开发的工作流

```
┌─────────────────────────────────────────────────┐
│  人：写规划文档（FINAL_PLAN.md）                   │
│       ↓                                          │
│  人：拆分任务（Step 1 → Step 10）                  │
│       ↓                                          │
│  人：给出当前 Step 的指令                          │
│       ↓                                          │
│  Agent：执行（写代码、改配置）                      │
│       ↓                                          │
│  人：验收（对照规划文档检查）                       │
│       ↓                                          │
│  人：git commit                                   │
│       ↓                                          │
│  （循环：下一个 Step）                             │
└─────────────────────────────────────────────────┘
```

## 注意事项清单

| # | 注意事项 | 为什么 |
|---|---------|--------|
| 1 | 规划文档必须自己写 | Agent 不能替你做产品决策 |
| 2 | 每步验收后再推进 | 早期发现错误，修复成本低 |
| 3 | 让 Agent 先读文档再写代码 | 避免 API 版本不匹配 |
| 4 | 类型定义先行 | 约束 Agent 的输出质量 |
| 5 | 频繁提交 | Git 是安全网 |
| 6 | 检查 Agent 的配置文件 | 它可能会遗漏关键配置 |
| 7 | 不要一次给太多指令 | 一步一步来，每步可控 |
| 8 | 保留 `CLAUDE.md` / `AGENTS.md` | 项目级指令，跨会话生效 |
